Introdução, Ferramentas e Boas Práticas

Visão completa da arquitetura Fluig, os 5 pilares da plataforma, e as convenções que separam projetos profissionais de projetos caóticos.

Fundação
Os 5 pilares do Fluig

Tudo no Fluig gira em torno desses 5 conceitos. Eles se conectam entre si — um formulário se vincula a um processo, que consulta um dataset, que alimenta um widget em um portal.

📄

Formulários

HTML puro com convenções Fluig. O atributo name dos campos é o que persiste os dados no banco.

⚙

Processos BPM

Workflows modelados em BPMN 2.0 — padrão internacional. Atividades, gateways, eventos e automações.

📊

Datasets

Funções que retornam dados em formato tabular. Podem buscar dados internos ou chamar APIs externas.

🌐

Portais / Pages

Páginas web gerenciáveis (WCM). Intranets, portais departamentais, landing pages internas.

🧩

Widgets

Componentes visuais reutilizáveis que se encaixam nos portais. Dashboards, gráficos, listas.

Arquitetura
Como o Fluig funciona por dentro

O Fluig é uma aplicação Java que roda dentro do WildFly (antigo JBoss). Tem 3 camadas distintas — saber onde cada código executa é a habilidade mais importante de um desenvolvedor Fluig.

Browser

Camada 1 — Cliente (navegador)

▾

Tudo que roda no navegador do usuário. Aqui você TEM acesso ao DOM, pode usar ES6+, jQuery, manipular elementos na tela.

O que roda aqui: Scripts dentro de tags <script> nos formulários, JavaScript de widgets, CSS do Style Guide, interações do usuário (cliques, validações visuais, máscaras de campo).

Engine: V8 (Chrome) ou SpiderMonkey (Firefox) — JavaScript moderno, sem restrições de versão.

Na prática: Quando você escreve um jQuery dentro do formulário para mascarar um CPF ou esconder uma div, isso roda aqui, no browser. Tem acesso a document, window, console, tudo.

↓ HTTP / REST ↓
Rhino ES5

Camada 2 — Servidor (Rhino JS)

▾

O Fluig usa o Mozilla Rhino, uma engine JavaScript escrita inteiramente em Java. Isso permite escrever JS que roda dentro do servidor Java, não no browser.

O que roda aqui: Datasets customizados (createDataset), eventos de formulário (displayFields, validateForm, beforeTaskSave), eventos de processo, callbacks.

Restrições críticas: Não tem document, window, console.log. Não tem let, const, arrow functions, template literals, async/await. Apenas var e function.

Superpoder: Acesso direto a classes Java! Pode usar java.net.URL para chamar APIs REST, java.util.HashMap para estruturas de dados, classes do SDK como hAPI e DatasetFactory.

Por que Rhino e não Node.js? Porque o Fluig roda dentro de um servidor Java (WildFly). O Rhino se integra nativamente com a JVM — pode chamar qualquer classe Java direto do JS. Node.js precisaria de uma ponte entre dois runtimes, adicionando complexidade. O trade-off é ficar preso ao ES5.

↓ JDBC / JPA ↓
Persistência

Camada 3 — Banco de dados + serviços Java

▾

A camada de persistência e serviços internos. Você nunca acessa o banco diretamente — o Fluig abstrai tudo via datasets e APIs internas.

Banco de dados: Suporta PostgreSQL, MySQL, Oracle e SQL Server. Armazena dados de formulários (cada campo name vira uma coluna), metadados de processos, documentos ECM, configurações.

SDK Java: hAPI movimenta processos automaticamente. DatasetFactory consulta datasets dentro de eventos. fluigAPI acessa serviços internos como usuários e grupos.

Solr: Motor de busca full-text (Apache Solr). Quando alguém pesquisa "nota fiscal" no Fluig, quem responde é o Solr — não uma query SQL. Ele indexa todos os conteúdos para buscas rápidas.

Conceito-chave: O atributo name de um campo HTML é o contrato entre o formulário e o banco. Quando você coloca name="fornecedor", o Fluig cria automaticamente uma coluna. Campo sem name = dado perdido para sempre.

Organização
Nomenclatura padrão JYNX

Com 200+ artefatos na plataforma, sem padrão vira caos. O prefixo identifica o tipo, o módulo agrupa por área de negócio.

Artefato Prefixo Padrão Exemplo
Formulário frm_ frm_[modulo]_[nome] frm_compras_solicitacao
Processo wkf_ wkf_[modulo]_[nome] wkf_compras_aprovacao
Dataset ds_ ds_[fonte]_[entidade] ds_protheus_fornecedores
Widget wdg_ wdg_[modulo]_[nome] wdg_compras_dashboard
Portal prt_ prt_[nome] prt_intranet
Estrutura de pastas no fluig Studio
projeto-fluig/
├── forms/              ← Formulários HTML
│   └── frm_compras_solicitacao/
│       ├── frm_compras_solicitacao.html
│       └── events.js   (eventos)
├── workflow/           ← Processos BPMN
│   └── wkf_compras_aprovacao/
├── datasets/           ← Datasets customizados
│   └── ds_protheus_fornecedores.js
├── wcm/                ← Portais e Widgets
│   ├── widget/
│   └── layout/
└── resources/          ← Imagens, CSS, JS
Regra de ouro: A pasta define o tipo do artefato. Um formulário dentro de forms/ faz deploy como formulário. Fora de forms/, o Studio ignora. Organização não é luxo — é requisito técnico.

Troubleshooting
Problema Causa provável Solução
Campo não salva dados Falta atributo name Adicionar name="campo" no input
Erro no dataset / evento Usando ES6 no servidor Usar apenas ES5 (var, function)
Identity vs Plataforma Produtos diferentes Identity = SSO. Plataforma = onde se trabalha
Dataset "not found" Nome errado (case sensitive) Verificar maiúsculas / minúsculas
Formulário sem estilo Falta wrapper do Style Guide Envolver com <div class="fluig-style-guide">
Não consigo logar Credenciais expiradas Solicitar reset ao administrador
