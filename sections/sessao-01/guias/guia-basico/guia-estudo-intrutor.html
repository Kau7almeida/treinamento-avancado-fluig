Domínio técnico
Arquitetura do Fluig — em profundidade

O Fluig na essência

O Fluig é uma plataforma corporativa da TOTVS que roda em cima de um servidor de aplicação Java chamado WildFly (antigo JBoss). Não é um produto único — é um ecossistema com 4 módulos integrados: ECM (documentos/formulários), BPM (workflows), WCM (portais/páginas), e Integração (datasets/APIs).

Todos os módulos rodam no mesmo servidor, no mesmo runtime Java. Não são microsserviços separados. Isso explica por que formulário conversa com processo, processo consulta dataset, e dataset alimenta widget — tudo está no mesmo "prédio".

Camada 1 — Browser (cliente)

Tudo que roda no navegador do usuário. Engine V8 (Chrome) ou SpiderMonkey (Firefox) — JavaScript moderno ES6+, acesso completo ao DOM.

O que executa aqui: Scripts dentro de <script> nos formulários, JavaScript de widgets, CSS do Style Guide, jQuery para manipulação de DOM, validações visuais, máscaras de campo.

Tem acesso a: document, window, console, DOM completo, localStorage, fetch API — tudo que o browser oferece.

Analogia: "É o JavaScript que vocês já conhecem. O mesmo que roda em qualquer site."

Camada 2 — Servidor (Rhino JS)

O Fluig usa o Mozilla Rhino, uma engine JavaScript escrita 100% em Java. O código JS roda dentro da JVM, no servidor — não no browser.

O que executa aqui: Datasets (createDataset), eventos de formulário (displayFields, validateForm, beforeTaskSave, etc), eventos de processo, callbacks.

NÃO tem acesso a: document, window, console, DOM, fetch, nada do browser. Só ES5 — sem let, const, arrow functions, template literals, async/await.

Superpoder: Acesso direto a classes Java! java.net.URL, java.util.HashMap, java.io.BufferedReader, e todo o SDK: hAPI, DatasetFactory, fluigAPI.

Analogia: "Existem dois mundos. O browser é colorido e moderno. O servidor é de 2010 — mas tem acesso a todo o poder do Java."

Camada 3 — Banco + serviços Java

Banco: PostgreSQL, MySQL, Oracle ou SQL Server. Cada campo name de um formulário vira uma coluna na tabela. O desenvolvedor nunca acessa o banco diretamente — tudo via datasets e APIs.

SDK: hAPI movimenta processos. DatasetFactory consulta datasets em eventos. fluigAPI acessa serviços internos. WCMAPI interage com portais.

Solr: Motor de busca full-text. Pesquisas no Fluig são processadas pelo Solr, não por queries SQL.

Por que Rhino e não Node.js?

1. Integração nativa com Java: Rhino é escrito em Java. Código JS instancia classes Java diretamente: var url = new java.net.URL("..."). Com Node.js, seria necessária uma ponte entre JVM e V8 — complexidade extra.

2. Sandbox controlado: O Rhino roda dentro da JVM com permissões restritas. Sem acesso a filesystem, sem sockets arbitrários. Segurança para ambiente multi-tenant.

3. Decisão histórica: Fluig foi arquitetado nos anos 2010, quando Java EE era o padrão corporativo. Migrar agora seria reescrever a plataforma.

4. Trade-off: Preso ao ES5, sem features modernas. Mas é estável e a integração com Java é o superpoder que justifica a limitação.

O atributo name — o contrato com o banco

Com name="fornecedor": O Fluig cria automaticamente uma coluna "fornecedor" na tabela do formulário. Dado persiste.

Sem name: O Fluig não sabe onde salvar. Dado some ao salvar. Erro #1 de iniciantes.

Com name duplicado: Segundo valor sobrescreve o primeiro. Silencioso e difícil de debugar.

Com caracteres especiais: Comportamento imprevisível. Usar sempre snake_case sem acentos: nome_completo, data_admissao.

Fluig Plataforma vs Identity: A Plataforma é onde tudo acontece (forms, processos, portais). O Identity é um produto separado de Single Sign-On. Analogia: Identity é o "porteiro" (quem é você?), Plataforma é o "prédio" (onde se trabalha). No treinamento, foco 100% na Plataforma.
Preparação para sala
Perguntas difíceis e como responder

Clique em cada pergunta para ver a resposta sugerida. Essas perguntas APARECEM em sala — esteja preparado.

FrequenteO Fluig é low-code ou high-code?
▾

É os dois. Para analistas: recursos low-code — processos BPM no editor visual, portais com drag-and-drop, Pages e Forms sem código. Para desenvolvedores: high-code — HTML/CSS/JS, datasets com lógica, integrações REST/SOAP, widgets avançados. O poder real aparece combinando as duas abordagens.

FrequenteDá para usar React/Vue/Angular?
▾

Tecnicamente é possível injetar em widgets, mas não é recomendado. O Fluig tem seu ecossistema (Style Guide, jQuery, componentes nativos). Misturar frameworks gera problemas de compatibilidade, performance e manutenção. Use Style Guide + jQuery — caminho suportado oficialmente.

FrequenteQual banco de dados recomendam?
▾

Suporta PostgreSQL, MySQL, Oracle e SQL Server. Depende da infra do cliente. Maioria usa SQL Server (por causa do Protheus) ou PostgreSQL. Para o desenvolvedor é irrelevante — nunca escreve SQL direto, tudo é abstraído por datasets.

FrequentePreciso saber Java?
▾

Não para o dia a dia. Você escreve JavaScript. Mas entender que existe Java por baixo ajuda a debugar e a usar classes Java quando necessário (ex: java.net.URL para integrações). O SDK expõe classes Java chamáveis do JS.

FrequenteO Fluig é cloud ou on-premise?
▾

Ambos. Pode rodar nos servidores da empresa ou na nuvem TOTVS. Arquitetura é a mesma — diferença é quem administra a infra. Para o desenvolvedor, o código é idêntico nos dois ambientes.

TécnicaO que acontece se usar let/const num dataset?
▾

Erro de sintaxe no servidor. O Rhino não entende ES6. O dataset não funciona — retorna erro ou null. O Fluig não valida a sintaxe JS no deploy, só quando executa. Então o deploy passa, mas o dataset falha silenciosamente na hora de usar.

Se não souber a resposta: "Boa pergunta — vou verificar na documentação oficial e trago a resposta confirmada na próxima sessão." Anote, pesquise, traga depois. O aluno respeita honestidade.
Técnica pedagógica
Erros propositais preparados

Mostrar o erro primeiro, deixar a turma diagnosticar, depois corrigir. Ensina debugging e ganha tempo de forma inteligente.

⚠ Erro 1: Campo sem name

Quando usar: ao explicar formulários e persistência (bloco dos 5 pilares)

O que fazer: Criar formulário com 2 campos. Primeiro com name="nome_completo". Segundo sem name (só id). Deployar, preencher ambos, salvar. Abrir registro — primeiro campo salvou, segundo está vazio.

"Viram? O primeiro campo salvou porque tem name. O segundo sumiu. O Fluig usa o name como endereço no banco. Sem endereço, o carteiro não sabe onde entregar."

Tempo ganho: 10-15 minutos de discussão + demonstração

⚠ Erro 2: ES6 no Rhino

Quando usar: ao explicar diferença browser vs servidor

O que fazer: Mostrar código com let, arrow function. Perguntar: "Se eu colocar isso num dataset, o que acontece?" Demonstrar o erro se tiver acesso ao ambiente, ou explicar com slide.

"O Rhino olha para esse let e não entende. É como falar inglês com alguém que só fala português. A sintaxe parece familiar, mas ele não reconhece."

Tempo ganho: 5-10 minutos

⚠ Erro 3: Confundir Plataforma com Identity

Quando usar: se alguém mencionar login, SSO, ou autenticação

O que fazer: Perguntar "Alguém sabe a diferença entre Fluig Plataforma e Fluig Identity?" Deixar a turma especular, depois explicar com analogia porteiro/prédio.

"O Identity é o porteiro — verifica quem é você. A Plataforma é o prédio — onde o trabalho acontece. São produtos diferentes que se integram."

Tempo ganho: 5-10 minutos de discussão

Memorizar
7 conceitos-chave

1. O Fluig é um monolito Java — todos os módulos (ECM, BPM, WCM) rodam no mesmo WildFly. Não são microsserviços.

2. Existem 2 mundos de JavaScript — browser (ES6+, DOM) e servidor (Rhino ES5, Java). Nunca confundir.

3. O atributo name é sagrado — sem ele, dado não persiste. Com duplicado, sobrescreve. Com acento, quebra.

4. Dataset não é tabela — é uma função que retorna dados tabulares. Pode vir do banco, API, ou cálculo.

5. BPMN 2.0 é padrão internacional — o conhecimento vale em qualquer ferramenta BPM, não só Fluig.

6. Style Guide é obrigatório — CSS customizado que conflita quebra em atualizações da plataforma.

7. Nomenclatura é requisito técnico — frm_, wkf_, ds_, wdg_, prt_. Sem padrão, caos.
