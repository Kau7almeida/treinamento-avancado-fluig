1.1 O servidor de aplicação: JBoss → WildFly

O que o WildFly faz pelo Fluig

  • Pool de conexões JDBC: Mantém conexões abertas com o banco. Datasets não abrem conexão nova — pegam do pool. Crítico para performance.
  • Gerenciamento de transações: Salvar formulário + mover processo + enviar notificação acontecem atomicamente — tudo passa ou tudo falha.
  • EJBs: Serviços do Fluig são EJBs. Quando o Rhino chama hAPI.moveTo(), invoca um EJB por baixo.
  • Classloader isolado: Bug em um dataset não derruba o servidor — o classloader isola o problema.
standalone.xml — o arquivo que controla tudo

Fica em WILDFLY_HOME/standalone/configuration/. Controla datasources, logging, pool de threads, segurança. O instrutor não edita, mas saber que existe impressiona a turma técnica.

standalone.xml 
<!-- Configuração de datasource (conexão com banco) -->
<subsystem xmlns="urn:jboss:domain:datasources:4.0">
    <datasource jndi-name="java:/jdbc/FluigDS" pool-name="FluigDS">
        <connection-url>jdbc:postgresql://localhost:5432/fluig</connection-url>
        <driver>postgresql</driver>
        <pool>
            <min-pool-size>10</min-pool-size>
            <max-pool-size>100</max-pool-size>
        </pool>
    </datasource>
</subsystem>
Se perguntarem "por que o Fluig tá lento?": Pode ser o max-pool-size muito baixo para o número de usuários. Conexões ficam na fila esperando.
1.2 Ciclo de vida de um deploy
1. Eclipse/Studio empacota artefatos em .war
2. Pacote enviado via HTTP para a API de deploy
3. Fluig descompacta no diretório de artefatos
4. Classloader carrega novos scripts/classes
5. Formulários HTML registrados no ECM
6. Processos BPMN registrados no BPM engine
7. Datasets JS registrados no registry
8. Solr reindexa novos conteúdos
9. Fluig invalida caches ← às vezes falha silenciosamente!
Problema comum: O passo 9 (invalidação de cache) falha silenciosamente. Por isso Ctrl+F5 no browser resolve muitos problemas pós-deploy.
1.3 Como o Fluig persiste dados de formulários

Quando um formulário é cadastrado, o Fluig lê o HTML, identifica os campos name, e cria uma tabela no banco com uma coluna para cada campo.

Tabela pai vs tabela filha (os 3 underscores)
HTML do formulário 
<!-- Campos da TABELA PAI (1 registro por formulário) -->
<input name="solicitante" type="text">
<input name="data_solicitacao" type="date">

<!-- Campos da TABELA FILHA (N registros — lista dinâmica) -->
<table>
  <tr>
    <input name="item_desc___1" type="text">   <!-- ___ = tabela filha -->
    <input name="item_qtd___1" type="number">
    <input name="item_valor___1" type="text">
  </tr>
</table>
No banco de dados 
Tabela principal (1 registro por formulário):
| documentid | solicitante  | data_solicitacao |
|------------|--------------|------------------|
| 12345      | João Silva   | 2026-03-17       |

Tabela filha (N registros por formulário):
| documentid | item_desc      | item_qtd | item_valor |
|------------|----------------|----------|------------|
| 12345      | Notebook Dell  | 2        | 4500.00    |
| 12345      | Mouse Wireless | 5        | 89.90      |
Os 3 underscores ___ são a convenção que o Fluig usa para identificar campos de tabela filha. O sufixo ___1, ___2 indica a linha. Crucial para listas dinâmicas nas sessões 03-04.

Rhino JS — limitações, workarounds e exemplos
Limitação 1: var vaza do escopo de bloco
// PROBLEMA: var vaza do for (ES5 não tem escopo de bloco)
for (var i = 0; i < items.length; i++) {
    var item = items[i];  // reatribuída a cada iteração
}
log.info("i ainda existe: " + i);  // funciona! não é undefined

// WORKAROUND: IIFE para criar escopo isolado
for (var i = 0; i < items.length; i++) {
    (function(index) {
        var item = items[index];  // cópia local, segura
        // lógica aqui dentro...
    })(i);
}
Limitação 2: sem template literals
// PROIBIDO:
var msg = `Olá ${nome}, pedido #${numero} aprovado`;

// CORRETO (concatenação simples):
var msg = "Olá " + nome + ", pedido #" + numero + " aprovado";

// WORKAROUND para strings longas (mais legível):
var msg = [
    "Olá ", nome, ",",
    " seu pedido #", numero,
    " foi aprovado em ", data,
    " pelo aprovador ", aprovador
].join("");
Limitação 3: sem map/filter nativos
// WORKAROUND: funções utilitárias reutilizáveis
function mapArray(arr, fn) {
    var result = [];
    for (var i = 0; i < arr.length; i++) {
        result.push(fn(arr[i], i));
    }
    return result;
}

function filterArray(arr, fn) {
    var result = [];
    for (var i = 0; i < arr.length; i++) {
        if (fn(arr[i], i)) { result.push(arr[i]); }
    }
    return result;
}

// Uso (quase tão limpo quanto ES6):
var nomes = mapArray(usuarios, function(u) { return u.nome; });
var ativos = filterArray(usuarios, function(u) { return u.ativo; });
Chamada REST com java.net.URL
// Dataset ou evento — roda no SERVIDOR (Rhino ES5)
function chamarApiRest(urlStr, metodo, authHeader) {
    var url = new java.net.URL(urlStr);
    var conn = url.openConnection();

    conn.setRequestMethod(metodo || "GET");
    conn.setRequestProperty("Content-Type", "application/json");
    conn.setRequestProperty("Accept", "application/json");

    // Basic Auth
    if (authHeader) {
        conn.setRequestProperty("Authorization", authHeader);
    }

    // Timeouts (milissegundos)
    conn.setConnectTimeout(10000);  // 10s para conectar
    conn.setReadTimeout(30000);     // 30s para ler resposta

    var responseCode = conn.getResponseCode();

    if (responseCode >= 200 && responseCode < 300) {
        var reader = new java.io.BufferedReader(
            new java.io.InputStreamReader(
                conn.getInputStream(), "UTF-8"
            )
        );
        var response = "";
        var line;
        while ((line = reader.readLine()) !== null) {
            response += line;
        }
        reader.close();
        return { status: responseCode, body: response, success: true };
    } else {
        return { status: responseCode, success: false };
    }
}

// Uso com Basic Auth do Protheus:
var auth = "Basic " + java.util.Base64.getEncoder().encodeToString(
    new java.lang.String("usuario:senha").getBytes("UTF-8")
);
var resultado = chamarApiRest("https://protheus.empresa.com/rest/SA2", "GET", auth);

    if (valorTotal > 10000) {
        // Mover para "Aprovação Diretoria" (atividade 3)

function beforeTaskSave(colleagueId, visaId, movementSequence) {
    var constraints = [];
    constraints.push(
        DatasetFactory.createConstraint(
            "codigo", "TI", "TI", ConstraintType.MUST
        )
    );
    var dataset = DatasetFactory.getDataset(
        "ds_departamentos", null, constraints, null
    );
    if (dataset.rowsCount > 0) {
        var gerente = dataset.getValue(0, "gerente");
        hAPI.setCardValue("aprovador", gerente);
    }
}

var WdgComprasDashboard = SuperWidget.extend({

        this.carregarDados();
    },

        local: {
            'click .btn-atualizar': ['carregarDados']

        DatasetFactory.getDataset(
            "ds_solicitacoes_pendentes", null, [], null,
            {
                success: function(data) {
                    self.renderizarTabela(data);
                },
                error: function(jqXHR, textStatus) {
                    self.showError("Erro: " + textStatus);
                }
            }
        );
    },
    renderizarTabela: function(data) {
        var html = '<table class="table table-striped">';
        html += '<thead><tr><th>ID</th><th>Solicitante</th></tr></thead>';
        html += '<tbody>';
        for (var i = 0; i < data.values.length; i++) {
            html += '<tr><td>' + data.getValue(i, "documentid")
                 + '</td><td>' + data.getValue(i, "solicitante")
                 + '</td></tr>';
        }
        html += '</tbody></table>';
        this.$el.find('.dashboard-content').html(html);
    }
});
Resolução de problemas
Cenários de debugging passo a passo

🔍 "O formulário salvou mas o campo está vazio"

Passo 1: Inspecionar HTML (F12)
  → O campo tem atributo "name"?
  → Se NÃO → adicionar name. Resolvido.

Passo 2: O name tem caracteres especiais?
  → Espaços, acentos, hífens?
  → Se SIM → trocar para snake_case sem acentos

Passo 3: O campo está dentro do <form>?
  → Verificar se não há </form> fechando antes do campo

Passo 4: Existe outro campo com o mesmo name?
  → Ctrl+F pelo valor do name no HTML
  → Se duplicado → segundo sobrescreve o primeiro

Passo 5: Algum evento JS está limpando?
  → displayFields escondendo pode limpar valor
  → Usar enableFields para desabilitar em vez de esconder

🔍 "O dataset retorna null ou vazio"

Passo 1: O dataset está registrado?
  → Admin → Datasets → Buscar (case sensitive!)
  → Não aparece → deploy não funcionou. Refazer.

Passo 2: Tem "return dataset;" no final?
  → Erro #1 mais comum.
  → Abrir código e verificar última linha.

Passo 3: Tem sintaxe ES6?
  → let, const, arrow function, template literal?
  → Rhino retorna null sem mensagem clara.

Passo 4: Chama API externa?
  → API acessível do SERVIDOR (não do browser)?
  → Testar: curl [URL] a partir do servidor

Passo 5: Constraint incorreto?
  → Campo inexistente → retorna vazio sem erro
  → Testar SEM constraints para isolar

🔍 "O processo trava — não avança"

Passo 1: Verificar Painel de Controle
  → Admin → Processos → Monitorar
  → Em qual atividade está parado?

Passo 2: Condições do Gateway
  → Gateway exclusivo: TODAS saídas precisam de condição
  → Nenhuma atendida = trava sem erro

Passo 3: Destinatário da próxima atividade
  → Tem destinatário definido? Usuário ativo?
  → Mecanismo de atribuição configurado?

Passo 4: Eventos com erro
  → beforeTaskComplete com exceção = não completa
  → hAPI.setAutomaticDecision com ID errado?

Passo 5: Log do servidor
  → WILDFLY_HOME/standalone/log/server.log
  → Buscar ScriptException ou ID do documento

🔍 "O Style Guide não funciona"

Passo 1: Tem o wrapper correto?
  → Precisa: <div class="fluig-style-guide">

Passo 2: Aberto dentro do Fluig?
  → Abrir file:// direto no browser não carrega Style Guide

Passo 3: CSS customizado conflitando?
  → style="" inline ou <style> pode sobrescrever

Passo 4: Versão do Fluig suporta o componente?
  → Componentes novos podem não existir em versões antigas

Se perguntarem sobre performance

▾

"O Fluig fica lento com muitos usuários. O que fazer?"

Verificar: max-pool-size no standalone.xml (pool de conexões), datasets sem constraint retornando milhares de registros, datasets jornalizados com intervalo muito curto, Solr desatualizado (re-indexar via admin), JVM com memória insuficiente (aumentar -Xmx no standalone.conf).

Se perguntarem sobre segurança

▾

"O que impede um usuário de ver dados que não deveria?"

4 níveis: Permissões ECM (quem vê cada documento), atividades BPM (só destinatário vê a tarefa), datasets (podem verificar usuário logado e filtrar), portais (widgets restritos por papel/grupo).

CUIDADO: displayFields esconde visualmente, mas o dado AINDA ESTÁ no HTML. Para segurança real, use permissões backend.

Se perguntarem sobre versionamento de código

▾

O Fluig não tem versionamento de código nativo (não é Git). Recomendação: usar Git externamente. O Studio exporta artefatos como arquivos — versione no Git. Para documentos ECM, o Fluig tem versionamento de documento (1.0, 2.0) — mas é para conteúdo, não para código.
